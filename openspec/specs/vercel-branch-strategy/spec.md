# vercel-branch-strategy Specification

## Purpose

Defines the branch-to-environment mapping that Vercel MUST apply when deploying this repository via the native GitHub Integration.

## Requirements

### Requirement: Production Environment Mapping

The `main` branch SHALL be the sole source for Production deployments. Every successful push to `main` MUST trigger a deployment to the Production environment and its stable production URL.

#### Scenario: Push to main deploys to production

- GIVEN a push is merged or pushed directly to `main`
- WHEN Vercel detects the push via GitHub Integration
- THEN Vercel MUST deploy to the Production environment
- AND the production URL MUST reflect the latest commit

### Requirement: Staging Environment Mapping

The `develop` and `staging` branches SHALL deploy to a Staging/Preview environment with a stable, non-ephemeral URL distinct from the production URL.

#### Scenario: Push to develop deploys to staging

- GIVEN a developer pushes to `develop`
- WHEN Vercel detects the push
- THEN Vercel MUST deploy to the Staging environment URL

### Requirement: Pull Request Preview Environments

Each pull request MUST receive an ephemeral Preview deployment with a unique URL generated by Vercel. The preview URL MUST be posted automatically to the pull request by Vercel's GitHub Integration.

#### Scenario: PR opened generates preview URL

- GIVEN a developer opens a pull request from any feature branch
- WHEN Vercel detects the new PR via GitHub Integration
- THEN Vercel MUST create an ephemeral deployment
- AND post the unique preview URL as a status check on the PR

### Requirement: GitHub Integration Prerequisite

The Vercel project MUST have GitHub Integration enabled in the Vercel Dashboard. Without this, `deployment_status` events will not fire and preview URLs will not be generated.
